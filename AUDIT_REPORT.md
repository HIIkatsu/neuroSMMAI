# Brutal Audit Report (NeuroSMMAI)

## Краткое резюме
Проект живой, но в текущем виде это **хрупкий монолит с элементами "работает — не трогай"**. Главные риски: неуправляемая сложность frontend-файла (`app.js` >5k строк), тихое проглатывание исключений по всему backend, слабая масштабируемость rate-limit и риск утечки Telegram initData в клиентском рантайме. Если пустить это в нагруженный прод без рефакторинга — получите нестабильные 500/429, трудноловимые баги и боль при расследовании инцидентов.

## Приоритизированный список проблем

| Приоритет | Локация | Проблема | Решение |
|---|---|---|---|
| Критический | `app.js`, `extractTelegramInitData()` и `api()` | **Небезопасная работа с Telegram initData на клиенте**: initData сохраняется в `sessionStorage`, затем дублируется и в `X-Telegram-Init-Data`, и в `Authorization: tma ...`. Любой XSS/инъекция в DOM = мгновенный захват сессии Telegram WebApp. Это плохой security posture для чувствительного аутентификационного материала. | Перевести auth-поток на одноразовый backend session exchange: `initData` отправляется 1 раз на `/api/auth/exchange`, сервер выставляет HttpOnly Secure cookie, дальше frontend не хранит initData вообще. Удалить fallback в `sessionStorage` и убрать дублирование в двух заголовках. |
| Критический | `auth.py`, `get_current_telegram_user()` | **Тихий fallback после ошибки Telegram auth**: `except HTTPException: pass` скрывает причину провала initData-валидации и тихо перескакивает на cookie flow. Это ломает наблюдаемость атак/ошибок и усложняет аудит. | Ловить только `401` от отсутствия данных, а при `Invalid hash/expired` логировать structured warning и возвращать 401 без fallback. Минимум: `except HTTPException as exc: if exc.detail == 'Missing Telegram init data': ... else: raise`. |
| Высокий | `miniapp_server.py`, `SimpleRateLimitMiddleware` | **Rate limit в памяти процесса** (dict+deque) — не масштабируется между воркерами/инстансами и легко теряет состояние при рестарте. На горизонтальном scaling получите непредсказуемые лимиты и abuse window. | Вынести лимитер в Redis (sliding window/token bucket). Ключ: `user_id` (если авторизован) + IP fallback. Добавить `Retry-After` и метрики (prometheus counters). |
| Высокий | `db.py` (множественные `except Exception: pass`) | **Говнокод уровня "заметаем ошибку под ковёр"**. Массовые пустые except'ы убивают дебаг и могут оставлять тихую порчу состояния/потери операций. | Заменить на узкие исключения + логирование с контекстом (`logger.exception`/`logger.warning`). В критичных местах — re-raise. Добавить линтер-правило на запрет bare/pass except. |
| Высокий | `app.js` (монолитный UI/logic файл) | **5k+ строк в одном файле — анти-SOLID и anti-DRY**: смешаны API-клиент, state machine, рендеринг, модалки, editor, assistant. Любая правка ломает смежный функционал. | Разбить на модули: `api/`, `auth/`, `state/`, `ui/modal/`, `features/editor/`, `features/assistant/`. Ввести typed contracts (TS или JSDoc + проверка), unit tests на чистые функции. |
| Средний | `app.js` (`innerHTML` по всему коду) | **Широкое использование `innerHTML`** при динамическом UI увеличивает XSS surface. Сейчас часть строк экранируется, но модель fragile: один забытый `escapeHtml` — и RCE в браузере. | Для пользовательских данных перейти на `textContent` + DOM API (`createElement`). Ввести helper-компоненты, где HTML-шаблоны принимают только заранее-санитизированные поля. |
| Средний | `miniapp_server.py`, middleware stack + CORS/CSRF | Логика защиты раздроблена между CSRF middleware и auth origin-check: часть путей допускает отсутствие `Origin/Referer`, часть — нет; поведение сложно предсказать и тяжело тестировать edge-cases. | Централизовать policy в одном security middleware с явной матрицей: `endpoint x auth_mode x method`. Добавить integration-тесты на spoofed origin, no-origin, Telegram headers only. |
| Средний | `app.js`, debug logging (`window._PREVIEW_DEBUG`, `console.log`) | В прод-коде оставлены debug-ветки с деталями media refs. Это шум, риск утечки приватных URL в клиентские логи и деградация UX на слабых девайсах. | Ввести compile-time/runtime флаг окружения, вырезать debug-логи из production bundle. |
| Низкий | `styles.css` и `miniapp/styles.css` | Дублирование одинаковых CSS-файлов = риск рассинхрона стилей и "fix in one place, broken in another". | Оставить один источник правды (`miniapp/styles.css`), второй файл заменить symlink/генерацией на этапе build/deploy. |
| Низкий | `styles.css` (`z-index: 100` на fixed nav) + модальные слои в JS | Потенциальный `z-index hell`: fixed nav и динамические backdrops из JS без централизованной шкалы слоёв. При доработках модалок легко поймать перекрытия/не-кликабельность в mobile WebView. | Ввести токены слоёв (`--z-nav`, `--z-modal`, `--z-toast`, `--z-overlay`) и единый layout contract. Добавить visual regression snapshot для модалок на мобильных ширинах. |

## Быстрый план стабилизации (первые 7 дней)
1. Закрыть auth-дыру по initData storage и fallback-логике.
2. Включить обязательное логирование всех auth-fail причин.
3. Перевести rate-limit в Redis.
4. Убрать пустые `except ...: pass` в критичных маршрутах и DB-слое.
5. Начать распил `app.js` на модули (минимум: auth/api/editor).
